# Flowlyst Website — system

**Status:** Approved in-session by Tural, 2026-07-12 (setup conversation). Amendments happen via issues in this repo.

## What this serves

A ground-up rewrite of **flowlyst.io** per [`docs/PRD.md`](../docs/PRD.md): 15 page templates (home, about, four solution pages, blog index + reader, contact, request-demo, testimonials, case-studies, and three legal pages), CMS-driven marketing content, and lead-generation forms. flowlyst is a US company serving K–12 public school districts with budgeting software, AI training, and AI/automation consulting.

The **old site** — `naysaziz/flowlyst-landing`, a Next.js + React-Admin monorepo on EC2/RDS — stays live and untouched in production until cutover. Nothing in this repo changes the legacy site.

## Operating model

The **main session runs on Fable 5** as orchestrator/architect: it plans, briefs, adjudicates review findings, and owns quality. It never bulk-produces artifacts. **All production work runs through Opus 4.8 subagents** defined in [`.claude/agents/`](../.claude/agents/): `coder`, `tester`, `code-reviewer`, `quality-engineer`, `env-ops`, `ui-verifier`. The full contract is in [`CLAUDE.md`](../CLAUDE.md).

## Stages (per work item)

```
pick issue → brief → build → test → independent review → quality gate → (ui-verify when anything visible changed) → merge → walkthrough note
```

## Gates

1. **Code review.** An independent, fresh-context `code-reviewer` agent reviews every change. Blocking — findings are fixed before merge. The reviewer judges; it never writes.
2. **Quality.** The `quality-engineer` builds, runs the full suite, and **exercises each acceptance criterion for real** (runs the dev server, hits routes, submits forms). No pass is ever reported that wasn't actually observed; unverifiable items are marked explicitly.
3. **UI evidence.** Any user-visible change requires `ui-verifier` screenshot verdicts — **light + dark, mobile (390px) + desktop (1440px), and empty/loading/error states.** UI is never called "done" from code inspection, and Tural is never asked to confirm what a screenshot can prove.
4. **Human gate — decisions and experience, not merge.** PRs **self-merge** once gates 1–3 pass. Every user-visible change lands on the **staging deployment** with a **walkthrough note** (what changed, the staging URL, what to try); Tural reviews by using it and his feedback becomes issues. His explicit word is required **before**: production/domain cutover, spending money, deleting anything outside this repo, any outward-facing act (sending emails, publishing), and any brand/positioning call not already settled in the PRD. **Vercel and Neon are his to operate** — agents never run those commands or touch those accounts; they prepare the config and a runbook, and Tural executes it.

## Autonomy

Overnight / unattended runs are the **intended mode**. An item blocked on a decision gets **parked with a decision note on its issue**, and work continues on unblocked items. The build does not stall waiting for Tural.

## Tracker

**GitHub Issues** on `flowlyst-io/flowlyst-website`. Items are written at outcome altitude. Every PR references its issue; `Closes #N` when the PR completes the item.

## Git

Trunk-based. Feature branches `feature/<issue>-<slug>`. **Squash merge.** `main` is always deployable to staging.

## Verification stack

- **Build** — the app builds clean.
- **Automated tests** — framework decided in-project when the first suite is written (recorded in the architecture-decision log, backlog issue 01).
- **Playwright screenshots** — the `ui-verifier` evidence surface (light/dark, mobile/desktop, state variants).
- **Lighthouse budgets + accessibility** (PRD §10, non-negotiable):
  - Lighthouse **Performance ≥ 90** on the homepage and each solution page (mobile).
  - **Largest Contentful Paint < 2.5s.**
  - **Cumulative Layout Shift < 0.1.**
  - **WCAG 2.1 AA** conformance: keyboard-navigable nav and forms, alt text on all imagery, visible focus indicators, one H1 per page. Corpowid accessibility widget preserved.

## Stack

- **Next.js — latest stable, App Router, on Vercel.** Decided by Tural, 2026-07-12 ("we're going to use Next.js' last version … App Router"). **TypeScript** throughout.
- **Every other technology choice is made in-project, agile** — decided as the work reaches it, not upfront, and recorded with a short rationale when a work item forces the call (backlog issue 01 is the running decision log). This covers the CMS, testing stack, form/email delivery, styling approach, and the ISR/revalidation strategy.
- **Neon Postgres** is the intended DB when one is warranted (e.g., form submissions, subscribers).
- **CMS** — decided in-project; whatever is picked must clear PRD §9's hard requirements (rich-text editor, media library, drafts, scheduled publishing, author profiles, preview, enforced Admin/Editor roles).
- **Vercel and Neon are Tural-operated.** Agents prepare config files, env-var lists, and migration scripts and hand Tural exact step-by-step runbooks; **they never run `vercel` / `neonctl` commands or touch those accounts.** (Tural's steer, 2026-07-12: "I will handle Vercel and Neon for now because it's dangerous stuff, especially after we go to production.")

  - **"flowlyst Website"** (id `019def0e-4193-726b-88e0-6ce28b50ecab`) — the complete hi-fi **page designs**; the reference each page implementation is checked against.
